第16周周报

一、项目实践内容
1. 前后端功能联调
本周开始了紧张的前后端联调工作。重点测试了“开始比赛 -> 计分 -> 结束比赛”的完整流程。修复了 API 路径不匹配的问题（如后端需要 `/match/start` 而前端误写为 `/start`）。修正了请求参数格式，确保前端发送的 JSON 数据能被后端正确解析。

2. 稳定性测试
对客户端进行了长时间运行测试，模拟了一场完整的网球比赛计分过程。重点检查了在网络波动情况下，计分请求是否会丢失，以及界面是否会卡顿。发现并修复了一个因未及时清理定时器导致的内存泄漏小问题。

3. 性能优化
针对视频上传慢的问题，优化了上传逻辑。虽然受限于网络带宽，但增加了上传进度提示，让用户知道系统在工作。优化了比分更新的响应速度，采用“乐观更新”策略，即点击按钮后立即更新界面数字，后台默默发送请求，提升了用户感知的流畅度。

二、实践收获
1. 掌握了联调技巧，学会了配合后端开发人员查看服务器日志，对比前端发出的请求和后端接收到的数据，快速定位问题。
2. 理解了“乐观更新”策略在提升用户体验方面的巨大作用，尤其是在网络延迟较高的情况下。
3. 学会了基本的性能分析方法，利用 Chrome DevTools 的 Performance 面板查看页面渲染性能和内存使用情况。

三、实践体会
联调是最考验耐心的阶段。很多时候是参数大小写、日期格式这种细节问题导致报错。通过这次联调，我深刻认识到接口文档准确性的重要性。如果文档写得清楚，能少走很多弯路。

四、建议
在联调前，前后端最好能坐在一起过一遍接口文档，确认每个字段的含义和格式。遇到问题多沟通，不要自己闷头猜。
